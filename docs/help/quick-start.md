# Quick Start

## What This App Does

Accord is a Windows wrapper around `accord.ps1`. It helps you run the cross-model QC pipeline without memorizing command-line flags or digging through session folders by hand.

The desktop app lets you:

- point the workspace at your target project directory
- run the pipeline in `code` or `document` mode
- monitor the active session
- resume a failed deliberation session from its saved session state
- inspect reports, logs, cycle files, and deliberation artifacts
- review a selected meta plan against the bundled checklist and save a sibling `*.reviewed.md`

## Before You Start

You need these tools installed and available on `PATH`:

- PowerShell
- Node.js
- Claude CLI
- Codex CLI
- Git

Use `Check Environment` in the app to confirm the desktop wrapper can see everything it needs.

## Where to Put the Plan File

The plan file belongs **in your target project folder**, not in the QC app folder.

> **Important:** The "Workspace Directory" in the app means your **target project directory** — the project you want Claude and Codex to work on (e.g., `C:\Projects\my-api`). It does **not** mean the QC app's own folder (`claude_codex_qc`).

The app resolves the plan file path relative to the workspace directory (your target project). If your workspace is `C:\Projects\my-api` and you enter `plan.md`, the app looks for `C:\Projects\my-api\plan.md`.

You can also use an absolute path to point anywhere, but relative paths always resolve against the target project directory.

All output (`logs/`, `cycles/`) is also written inside the target project directory.

## First Run

1. Launch the desktop app.
2. Set `Workspace Directory` to your **target project directory** — the project you want Claude and Codex to work on (not the QC app folder).
3. Set `Plan File` to the markdown plan that describes the task. This file should live inside (or relative to) the target project directory.
4. Confirm the `Prompt Directory` if you need a custom prompt pack.
5. Choose `QC Type`.
6. Leave advanced settings alone unless you know you need different iteration limits.
7. Click `Start Run`.

## QC Type

The `QC Type` dropdown controls what kind of output the pipeline produces and how it is reviewed.

- **code** (default): Use when the end result is source code. Claude writes code from your plan, Codex reviews it for correctness, security, performance, and error handling.
- **document**: Use when the end result is a plan or written document. Claude generates a document, Codex reviews it for completeness, clarity, structure, and actionability.

| Scenario | QC Type | Why |
|----------|---------|-----|
| Implementing a feature or bug fix | `code` | Standard code QC |
| Generating a cycle plan from a meta-plan | `document` | Focus on document quality |
| Creating documentation or specifications | `document` | Focus on completeness and clarity |
| Mixed code + docs task | `code` | Code mode as baseline |

## Flags

Three toggle switches in the Configure panel control pipeline behavior:

### Deliberation Mode

Replaces the standard write-review-fix loop with a structured multi-round collaboration. Claude implements and documents its thinking in a thoughts file; Codex evaluates, makes direct edits to the output, and declares a decision. Claude then reviews the feedback and refines if needed. This continues until both agents converge.

The orchestrator enforces the convergence gate: if any CRITICAL or HIGH issues remain in Codex's `Remaining Issues` section, CONVERGED is vetoed and another round is forced — even if Codex declares CONVERGED.

Use this for ambiguous tasks, architecture decisions, or when quality matters more than speed.

### Skip Plan QC

Skips Phase 0 (plan validation) and jumps straight to implementation. Use this when your plan is already trusted — for example, a continuation cycle plan generated by a previous run, or a plan you have already reviewed and validated manually.

### Pass On Medium Only

Normally the pipeline requires zero CRITICAL and zero HIGH severity issues to pass. With this flag enabled, the pipeline exits early once only MEDIUM or LOW issues remain, saving iterations and token cost. Use this when minor code quality issues are acceptable and you want faster turnaround.

## Common Examples

```powershell
# Basic code implementation (default settings)
.\accord.ps1 -PlanFile "plan.md"

# Document mode — generate a detailed plan from a meta-plan
.\accord.ps1 -PlanFile "meta_plan.md" -QCType document

# Skip plan review for a trusted continuation cycle
.\accord.ps1 -PlanFile "cycles\<session_id>\CONTINUATION_CYCLE_1.md" -SkipPlanQC

# Deliberation mode for complex or ambiguous work
.\accord.ps1 -PlanFile "plan.md" -DeliberationMode

# Document deliberation — both LLMs collaborate on plan generation
.\accord.ps1 -PlanFile "meta_plan.md" -QCType document -DeliberationMode

# Lenient pass criteria + skip plan QC for fast iteration
.\accord.ps1 -PlanFile "plan.md" -SkipPlanQC -PassOnMediumOnly

# Hybrid QC — Claude does first 3 iterations, then Codex takes over
.\accord.ps1 -PlanFile "plan.md" -ClaudeQCIterations 3
```

## Creating a Meta-Plan in the App

Click the **+ New** button next to the Plan File field. This opens a built-in editor in the viewer pane with a ready-to-use meta-plan template. Fill in your goal, features, and constraints, choose a file name (defaults to `meta_plan.md`), and click **Save to Project**. The file is saved into your target project directory and automatically set as the active plan file.

After saving, select `document` as QC Type, optionally enable Deliberation Mode, and click Start Run.

If you already have a sparse meta plan and want to strengthen it first, click **Review Meta Plan** instead of Start Run. That button:

- uses the selected Plan File as the source meta plan
- compares it against the bundled `plan_for_meta_plan.md` checklist
- writes a sibling reviewed file such as `meta_plan.reviewed.md`
- leaves the original selected file unchanged

## Two-Stage Workflow: From Idea to Code

This is the most powerful workflow in the pipeline. Both LLMs collaborate to turn a rough idea into working code in two stages.

### Stage 1: Generate a detailed plan

Write a short meta-plan describing what you want at a high level (or use the **+ New** button to start from a template):

```markdown
# Meta-Plan: Build a Rate Limiter

## Goal
Create Express middleware that limits API requests per IP using Redis.

## Features
1. Sliding window algorithm
2. Configurable window and max requests
3. Returns 429 with Retry-After header

## Constraints
- Must handle Redis connection failures gracefully
- TypeScript with strict mode
```

Then run document mode (optionally with deliberation) to generate a detailed implementation plan:

```powershell
# Standard document QC — Claude writes plan, Codex reviews, loop until pass
.\accord.ps1 -PlanFile "meta_plan.md" -QCType document

# Or with deliberation — Claude and Codex collaborate back-and-forth
.\accord.ps1 -PlanFile "meta_plan.md" -QCType document -DeliberationMode
```

The output is a `CONTINUATION_CYCLE_*.md` file in `cycles/<session_id>/` — a detailed, QC-validated implementation plan with file paths, function signatures, edge cases, and test cases.

Document deliberation enforces four non-negotiable sections in the output: **Quickstart Tutorial** (prerequisites, exact commands), **Frontend Design Notes** (UI cycles only), **QC Lessons Learned**, and **Next Cycle Instructions**. If any are missing, Codex marks a CRITICAL issue and the pipeline forces another round.

### Stage 2: Implement code from the plan

```powershell
.\accord.ps1 -PlanFile "cycles\<session_id>\CONTINUATION_CYCLE_1.md" -SkipPlanQC
```

The full pipeline:

```
Rough Idea → meta_plan.md → Document QC/Deliberation → CONTINUATION_CYCLE_N.md → Code QC → Working Code
```

## Severity Levels and Pass Criteria

| Severity | Definition | Blocks pass? |
|----------|------------|-------------|
| **CRITICAL** | Security vulnerability or data loss risk | Yes |
| **HIGH** | Incorrect behavior, crash, or logic error | Yes |
| **MEDIUM** | Code quality or maintainability issue | No |
| **LOW** | Style or minor improvement | No |

**Default:** QC passes when there are zero CRITICAL and zero HIGH issues.

**With Pass On Medium Only:** Same criteria, but the pipeline stops early once only MEDIUM/LOW issues remain instead of trying to fix them.

## Common Workflow

- `Start Run`: launches a new session with the current form settings.
- `Resume Selected`: resumes the selected failed, stale, or cancelled deliberation session from the last unfinished step.
- `Cancel Selected`: stops the selected running session.
- `Refresh`: reloads the session list from `logs/` and `cycles/`.

## Resuming After Failure or App Restart

Resume is tied to the selected session, not to the current form values.

1. Set `Workspace Directory` to the same target project directory where the run failed.
2. Refresh or wait for the session list to load.
3. Select the failed deliberation session.
4. Click `Resume`.

Important behavior:

- Resume ignores the current Configure form values and uses the selected session's saved `session_config.json`.
- Resume keeps the same `session_id` and continues from the last saved deliberation point.
- Resume is currently available for `document` and `code` deliberation sessions only.

## Where Output Goes

Each run writes into a session-scoped layout inside your target project directory:

- `logs/<session_id>/`
- `cycles/<session_id>/`

Typical files include:

- `run_state.json`
- `events.ndjson`
- QC reports
- plan QC reports
- `_iteration_history.md`
- `CONTINUATION_CYCLE_*.md`

For deliberation sessions, a `deliberation/` subfolder holds round-by-round artifacts:

```
deliberation/
├── phase0/                          # document deliberation
│   ├── round1_claude_thoughts.md
│   ├── round1_codex_evaluation.md
│   └── deliberation_summary.md      # written at convergence
└── phase1/                          # code deliberation
    ├── round1_claude_thoughts.md
    ├── round1_codex_review.md
    └── deliberation_summary.md
```

## Reading Results

- Use the `Session Rail` to select a session.
- Use `Session Files` to open a report or cycle file in the viewer.
- Use `Recent Events` to see the structured event stream.
- Use `Open Externally` if you want the current file in your editor or default app.

## If Something Fails

- Run `Check Environment` and fix anything marked missing.
- Make sure your plan file exists and is readable.
- Confirm Claude CLI and Codex CLI are authenticated outside the app.
- Check the session log and recent events for the failure message.
- If the failed run was a deliberation session, select it and use `Resume` instead of `Start Run`.
